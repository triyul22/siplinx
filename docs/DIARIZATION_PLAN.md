# План: кластеризация голосов (speaker diarization) — будущая PRO-фича

Статус: **отложено, не реализуем сейчас.** Документ — план на потом.
Цель: «кто что сказал» с разделением и переименованием спикеров. Заводится под `ProGate` как PRO-функция.

## Контекст (что есть сейчас)
- Приложение делит транскрипт **по источнику**: `mic` (ты) vs `system` (звонок/остальные) — миграция `add_speaker_field.sql`.
- Различения отдельных голосов внутри потока **нет**. Поле `speaker_embedding` в `stt.rs` — пустой задел.
- Стек: Tauri 2 + Rust + whisper.cpp (whisper-rs) + `silero_rs` (VAD). Privacy-first, всё локально.

## Пайплайн диаризации (4 шага)
1. **VAD** — сегменты речи. Уже есть `silero_rs`.
2. **Speaker embeddings** — вектор «отпечаток голоса» (~192–256 dim) на каждый сегмент (ECAPA / WeSpeaker / 3D-Speaker).
3. **Кластеризация** — группировка по косинусной близости. Неизвестное число спикеров → AHC с порогом; известное → задаём N.
4. **Выравнивание** — присвоить спикера каждому сегменту whisper по перекрытию по времени.

## Архитектурные решения под наш проект
- **Батч, не реалтайм.** Запускать диаризацию ПОСЛЕ остановки записи на сохранённом WAV (`recording_saver.rs`), затем переразмечать транскрипт. Онлайн-диаризация — позже, сложнее и хуже.
- **Диаризовать только `system`-поток.** Микрофон всегда «ты»; несколько голосов идут из `system`. Меньше данных, выше точность.
- **Число спикеров:** авто (порог кластеризации) + возможность задать/подсказать N в UI.

## Технологии (выбор)
**Рекомендация: `sherpa-rs`** (Rust-биндинги к sherpa-onnx).
- Умеет сегментацию + эмбеддинги + кластеризацию, офлайн, ONNX Runtime, Win/mac/Linux, CPU/GPU.
- Модели качаются как whisper-модели.

Альтернативы:
- **`speakrs`** — чистый Rust, пайплайн pyannote community-1 (сегментация + эмбеддинги + VBx-кластеризация) на ONNX. Минимум C++.
- **pyannote.audio (Python)** — макс. точность, но PyTorch в опциональном backend (тяжелее, ломает «чистый локальный десктоп»). Только если точность критична.
- **whisper.cpp tinydiarize — НЕ подходит:** ловит только смену говорящего, не кластеризует, моно.

## Точки интеграции (конкретно)
- Новый модуль `frontend/src-tauri/src/audio/diarization.rs` на `sherpa-rs`: WAV → VAD → эмбеддинги → AHC → сегменты со спикерами + таймкоды.
- Выравнивание с `TranscriptionResult.start_time/end_time` (`audio/stt.rs`) по max-overlap.
- Хранение: переиспользовать колонку `speaker` или добавить `speaker_cluster`; отдельная таблица имён для переименования «Спикер 1» → «Анна».
- UI: метки спикеров + переименование, всё внутри `ProGate` (`frontend/src/components/auth/ProGate.tsx`).
- Запуск: триггерить после остановки записи (post-processing), показывать прогресс.

## Модели (sherpa-onnx model zoo → `frontend/models/`)
- Сегментация: `sherpa-onnx-pyannote-segmentation-3.0`
- Эмбеддинги: `3dspeaker_eres2net` или `nemo_titanet_small` (~десятки МБ)

## Оценка и риски
- Батч-MVP (только `system`, авто-N, метки + переименование, под `ProGate`): **~3–5 дней**, основную тяжесть берёт sherpa.
- Точность падает на перекрывающейся речи, похожих голосах, коротких репликах — норма для диаризации.
- Завязать на сервер (а не только клиент), если хотим, чтобы фичу нельзя было разблокировать форком (репо MIT — клиентский гейт обходится).

# Как формируется саммари (Siplinx AI)

Документ описывает текущую логику генерации саммари встречи: модель, последовательность шагов и точные промпты.
Код: `frontend/src-tauri/src/summary/processor.rs` (+ `summary_engine/` для локальной модели).

## Модель и провайдер

- **Дефолт: `builtin-ai` = Gemma 3 1B** (Q8_0 GGUF, ~1 ГБ), работает локально через bundled-сайдкар `llama-helper.exe` (без интернета, без Ollama).
- Транскрипция — отдельная модель (Parakeet/Whisper), не входит в саммари.
- Сэмплинг (в `summary_engine/models.rs`): `temperature 0.3, top_k 40, top_p 0.9`. На шаге extract температура форсится в **0.2** (грунтованный JSON).
- Альтернативы (если пользователь выбрал): Ollama / OpenAI / Claude / Groq — облачный путь (см. ниже).

## Вход

1. Стенограмма = склейка текстов всех сегментов из `transcripts.json`.
2. **Детект языка** (`transcript_is_cyrillic`): если ≥25% букв кириллица → встреча считается русской. От этого зависит язык промптов. Это фикс «саммари на английском»: маленькая модель пишет на языке промпта, поэтому для русской встречи промпты на русском.

## Локальный путь (Gemma 1B) — цепочка из 3 шагов

```
стенограмма
  └─ режем на чанки ~1800 токенов (overlap 200)
       └─ ШАГ 1 EXTRACT (модель, на каждый чанк): извлечь факты → строгий JSON
            └─ ШАГ 2 MERGE (Rust-код, без модели): распарсить JSON, дедуп, выкинуть мусор
                 └─ ШАГ 3 COMPOSE (модель, 1 раз): собрать отчёт из чистых заметок
                      └─ финальный Markdown → сохраняется в SQLite
```

Почему так: на 1 большой промпт «сделай саммари» слабая модель даёт воду/выдумки/английский. Узкие однозадачные промпты + дедуп в коде дают на порядок чище.

### Шаг 1 — EXTRACT (промпт для русской встречи)

Системный промпт (на русском, с **плейсхолдер-примером** — конкретные имена в примере провоцируют 1B их копировать/выдумывать):

```
Ты составляешь протокол встречи на русском языке. Верни СТРОГО один JSON-объект
(без текста вокруг). Все значения — по-русски.

Схема: {"decisions":[строки], "action_items":[{"owner":строка,"task":строка,"due":строка}],
"questions":[строки], "key_points":[строки], "participants":[строки]}

Правила:
- Бери ТОЛЬКО то, что ЯВНО сказано в стенограмме. Не выдумывай.
- participants: только имена конкретных людей. НЕ включай отделы, роли, числа. Нет имён — [].
- action_items: только явные поручения. Сомневаешься — не добавляй.
- Лучше пустой список [], чем выдуманный пункт.
- Фрагменты, искажённые распознаванием речи, игнорируй.
- Пример ниже показывает ТОЛЬКО ФОРМАТ. НЕ копируй из него ничего.

ПРИМЕР ФОРМАТА (заполнители, не данные):
{"decisions":["<решение>"],"action_items":[{"owner":"<имя>","task":"<задача>","due":"<срок>"}],
"questions":["<вопрос>"],"key_points":["<тема>"],"participants":["<имя>"]}
```

Пользовательский промпт: `<transcript>{чанк}</transcript>`. Температура 0.2.
(Для английской встречи — аналогичный промпт на английском.)

### Шаг 2 — MERGE (чистый Rust, без модели)

- `parse_chunk_facts`: лениво вытаскивает внешний `{...}` (режет ```-фенсы и прозу), парсит в `ChunkFacts`. Если ни один чанк не дал валидный JSON — fallback на склейку сырых extract'ов (не хуже прежнего).
- `merge_facts` + `dedup_preserve_order`: объединяет списки по всем чанкам, дедуп (кейс-инсенситив, сохраняет порядок), выкидывает мусор `is_junk` (пустые, `[]`, `].`, «-», только-пунктуация).
- `to_display` для задач: `owner: task (due)`, но junk-owner/`«не указано»`-due отбрасываются.
- `render_notes`: рендерит чистые заметки с заголовками на языке встречи (РЕШЕНИЯ/ЗАДАЧИ/ОБСУЖДЕНИЕ/ОТКРЫТЫЕ ВОПРОСЫ/УЧАСТНИКИ).

### Шаг 3 — COMPOSE (промпт для русской встречи)

Для локальной модели **шаблон встречи (с таблицами) НЕ используется** — 1B галлюцинирует колонки. Вместо него фиксированный промпт:

```
Ты составляешь итоговый протокол встречи на русском языке по готовым заметкам. Пиши ТОЛЬКО по-русски.
Используй ТОЛЬКО факты из заметок ниже. Ничего не добавляй и не выдумывай.

Начни с заголовка `# <короткое название встречи, 3-6 слов>`. Затем разделы Markdown:
## Краткое содержание (2-4 предложения)
## Ключевые решения (список; если нет — «Не зафиксировано»)
## Задачи (список «исполнитель: задача (срок)»; если нет — «Не зафиксировано»)
## Обсуждение (список главных тем)
## Открытые вопросы (список; если пусто — пропустить раздел)

Без таблиц. Без вступления и заключения. Только сам отчёт.
```

Пользовательский промпт: `<заметки>{чистые заметки из шага 2}</заметки>`.
Заголовок `# ...` затем вытаскивается как название встречи (`extract_meeting_name_from_markdown`).

## Облачный путь (OpenAI/Claude/Groq/Ollama-выбор)

Один проход: берётся весь транскрипт + выбранный пользователем шаблон (`standard_meeting` / `daily_standup`), модель заполняет его. Цепочка extract→merge не применяется (большие модели тянут один промпт).

## Ключевые места в коде

- `processor.rs::generate_meeting_summary` — оркестратор всей цепочки.
- `processor.rs` константы: `LOCAL_EXTRACT_CHUNK_TOKENS=1800`, `LOCAL_EXTRACT_OVERLAP_TOKENS=200`, `RU_COMPOSE_PROMPT`/`EN_COMPOSE_PROMPT`.
- `processor.rs`: `ChunkFacts`, `merge_facts`, `dedup_preserve_order`, `is_junk`, `transcript_is_cyrillic`, `render_notes`.
- `summary_engine/models.rs` — модель и сэмплинг. `llama-helper/src/main.rs` — сам сайдкар (JSON по stdin/stdout).

## Известные ограничения (потолок качества)

1. **ASR (Parakeet) плохо распознаёт русский** → грязная стенограмма → часть «выдумок» = ошибки распознавания. Самый большой рычаг: перевести русскую транскрипцию на Whisper large-v3.
2. **1B физически слаб** — даже с чистыми промптами не уровень GPT-4. Кандидаты усиления: GBNF grammar (гарантия валидного JSON, нужна доработка сайдкара), смена модели (Qwen3 1.7B), repeat-penalty в сайдкаре.
